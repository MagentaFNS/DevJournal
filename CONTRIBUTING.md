# Руководство по внесению вклада в DevJournal

Привет! 👋 Мы рады, что ты хочешь помочь сделать **DevJournal** лучше. Этот проект — попытка превратить ежедневные отчеты команды в полезную аналитику, и мы верим, что вместе сможем создать действительно классный инструмент. Независимо от твоего опыта — ты можешь помочь кодом, документацией, идеями или сообщениями об ошибках.

Этот документ — наш общий договор о том, как мы будем работать вместе. Пожалуйста, внимательно ознакомься с ним, прежде чем начать.

## Наша философия

DevJournal создается для разработчиков и команд, которые хотят рефлексировать и видеть свой прогресс. Мы придерживаемся следующих принципов:

*   **Простота**: Проект должен быть понятным и легким в настройке.
*   **Полезность**: Каждая функция должна приносить реальную ценность пользователю.
*   **Открытость**: Мы открыты для идей и критики, весь код и обсуждения публичны.

## Как я могу помочь?

Любая помощь важна! Вот несколько направлений:

### Сообщение об ошибке

Если ты нашел баг, создай **Issue** в нашем репозитории. Пожалуйста, убедись, что такой проблемы еще нет в списке. В описании постарайся ответить на вопросы:
1.  Что ты делал?
2.  Что ожидал увидеть?
3.  Что произошло на самом деле?
4.  Если есть, приложи скриншоты или логи.

### Предложение новой идеи

У тебя есть идея, как сделать DevJournal лучше? Отлично! Создай **Issue** с пометкой `enhancement` и подробно опиши:
*   В чем суть идеи?
*   Какую проблему она решает?
*   Как, по-твоему, это должно работать?

### Улучшение документации

Хорошая документация — это основа проекта. Ты можешь:
*   Исправить опечатки или неточности в `README.md`.
*   Дополнить руководство по установке.
*   Написать примеры использования.
*   Улучшить этот файл (`CONTRIBUTING.md`).

## Начало работы с кодом

### Настройка окружения

Процесс описан в главном `README.md`, но дублируем его для ясности:

```bash
# 1. Клонируй репозиторий
git clone https://github.com/MagentaFNS/DevJournal.git
cd DevJournal

# 2. Создай и активируй виртуальное окружение
# Для macOS/Linux:
python -m venv venv
source venv/bin/activate

# Для Windows (PowerShell):
python -m venv venv
venv\Scripts\activate

# 3. Установи зависимости
pip install -r requirements.txt
```
### Структура проекта

Чтобы тебе было проще ориентироваться, вот основные папки:

desktop/ - код десктопного приложения.
mobile/ - код мобильного приложения.

docs/ - документация.

### Процесс работы над задачами
## Поиск задачи

Загляни в раздел Issues на GitHub.
Ищи задачи с пометкой good first issue — они идеально подходят для новичков.
Если хочешь взяться за задачу, напиши об этом в комментариях, чтобы мы знали, что никто не работает над ней параллельно.

## Создание ветки

Создай ветку от актуального main (или master). Назови ее осмысленно, например:
feature/add-commit-graph — для новой функциональности.
bugfix/fix-auth-error — для исправления бага.
docs/update-readme — для изменений в документации.

### Соглашения о коммитах
Мы стараемся придерживаться Conventional Commits. Это помогает автоматически генерировать changelog и лучше понимать историю изменений.

Основной формат:
<type>(<scope>): <subject>
type: feat (новая фича), fix (багфикс), docs (документация), style (форматирование), refactor (рефакторинг), test (тесты), chore (служебные задачи).
scope: необязательная часть, указывает, что именно изменилось (например, backend, website).
subject: краткое описание изменений в настоящем времени.

Примеры:

feat(backend): add endpoint for daily report generation
fix(website): resolve chart rendering issue on Safari
docs: update installation guide for Windows

## Создание Pull Request (PR)
Когда твоя работа готова, создай Pull Request из своей ветки в main.

В названии PR кратко опиши суть изменений.
В описании PR подробно расскажи:
Какую проблему решает? (укажи номер Issue, если есть: Closes #123).
Что было сделано? Перечисли ключевые изменения.
Как проверить? Опиши шаги для тестирования твоих изменений.
Дождись ревью от других участников проекта. Будь открыт к комментариям и готов доработать свой код.

### Стандарты кода
Чтобы код был единообразным, мы используем следующие инструменты:

Python (бэкенд): Следуй стандарту PEP 8. По возможности, используй black для автоформатирования.
JavaScript (фронтенд): Используй Prettier и ESLint. Конфигурационные файлы уже есть в проекте.
Перед созданием PR обязательно проверь свой код на наличие стилистических ошибок.

### Тестирование
Мы ценим стабильность. Если ты добавляешь новую функциональность или исправляешь баг, пожалуйста:

Убедись, что существующие тесты проходят.
Напиши новые тесты для покрытия твоих изменений, если это возможно.
Мы используем (укажи свою систему тестирования, например pytest для Python). Инструкции по запуску тестов будут добавлены в README.md позже.

### Лицензия
